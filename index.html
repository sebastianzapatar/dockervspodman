Antes de Docker o Podman

El problema

  • “En mi máquina sí funciona”.
  • Diferencias de versiones entre ambientes.
  • Instalaciones manuales difíciles de replicar.
  • Conflictos entre dependencias.

La solución

  • Empaquetar app + dependencias.
  • Ambientes reproducibles.
  • Ejecución aislada.
  • Despliegues más consistentes.

Conceptos base

Imagen

Plantilla inmutable con sistema de archivos, dependencias y comando de arranque.

Contenedor

Proceso aislado creado a partir de una imagen.

Registry

Repositorio donde se publican y descargan imágenes.

Volumen

Persistencia de datos fuera del ciclo de vida del contenedor.

Red

Comunicación controlada entre contenedores y con el exterior.

Runtime

Componente que ejecuta el contenedor según estándares como OCI.

OCI: el estándar común

Docker y Podman se apoyan en estándares de la Open Container Initiative, que define especificaciones abiertas para formatos de imagen y runtimes de contenedores.

Docker: arquitectura

Cómo trabaja

  • Cliente docker.
  • Daemon dockerd.
  • Imágenes y contenedores gestionados por el motor.
  • Integración fuerte con Docker Desktop y Docker Compose.

Fortalezas

  • Ecosistema muy amplio.
  • Documentación y comunidad maduras.
  • Excelente para iniciar en contenedores.
  • Soporte robusto para Compose.

Podman: arquitectura

Cómo trabaja

  • Modelo daemonless.
  • Ejecución rootless como caso de uso importante.
  • CLI muy similar a Docker.
  • Soporte nativo para pods.

Fortalezas

  • Menor dependencia de un daemon central.
  • Buen enfoque de seguridad local.
  • Integración con systemd en Linux.
  • Puente interesante hacia Kubernetes.

Docker vs Podman

CriterioDockerPodman
ArquitecturaCliente-servidor con daemon.Daemonless; procesos gestionados por usuario.
RootlessDisponible, pero no siempre es el flujo por defecto.Parte central de su propuesta.
ComposeMuy maduro y ampliamente usado.Compatible mediante Podman Compose o Docker API socket.
PodsNo es su abstracción principal.Soporte nativo para pods.
Uso docenteIdeal para iniciar y por disponibilidad de recursos.Ideal para discutir seguridad, rootless y Kubernetes.

¿Cuál usar?

Usar Docker cuando...

  • El equipo apenas inicia.
  • Se necesita compatibilidad amplia con tutoriales.
  • Se usa mucho Docker Desktop.
  • Se requiere Compose sin fricción.

Usar Podman cuando...

  • Se quiere trabajar rootless.
  • Se prioriza seguridad local.
  • Se trabaja en Linux empresarial.
  • Se quiere acercar el flujo a pods/Kubernetes.

Decisión de curso: Docker como base; Podman como alternativa profesional y puente conceptual.

Dockerfile / Containerfile

Un Dockerfile es un documento de texto con instrucciones para construir una imagen. En Podman suele llamarse también Containerfile, aunque puede usar Dockerfile sin problema.

# Etapa de construcción
FROM node:22-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Etapa de ejecución
FROM node:22-alpine
WORKDIR /app
ENV NODE_ENV=production
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/dist ./dist
EXPOSE 3000
CMD ["node", "dist/index.js"]

Buenas prácticas de imagen

Recomendado

  • Usar imágenes base pequeñas y oficiales.
  • Aplicar multi-stage build.
  • Copiar primero archivos de dependencias.
  • Usar .dockerignore.
  • No subir secretos a la imagen.

Evitar

  • Ejecutar como root si no es necesario.
  • Instalar herramientas innecesarias.
  • Usar etiqueta latest sin control.
  • Guardar datos dentro del contenedor.
  • Mezclar build y runtime sin necesidad.

Redes y volúmenes

Redes

  • Permiten comunicación entre servicios.
  • En Compose, los servicios se resuelven por nombre.
  • Evitan publicar todos los puertos al host.

Volúmenes

  • Persisten datos aunque el contenedor se elimine.
  • Clave para bases de datos.
  • Separan aplicación de estado.
docker network create app-net
docker volume create pgdata
docker run -d --name postgres --network app-net -v pgdata:/var/lib/postgresql/data postgres:16

Compose: varios servicios

services:
  api:
    build: .
    ports:
      - "3000:3000"
    environment:
      DB_HOST: db
      DB_USER: postgres
      DB_PASSWORD: postgres
      DB_NAME: appdb
    depends_on:
      - db

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: appdb
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
    volumes:
      - pgdata:/var/lib/postgresql/data

Health checks

Un contenedor puede estar “corriendo”, pero la aplicación no necesariamente está lista. Por eso se recomienda exponer endpoints como /health y /ready.

HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1

Seguridad mínima

Contenedores

  • No ejecutar procesos como root si no es necesario.
  • No montar el socket de Docker sin justificación.
  • Usar imágenes oficiales o verificadas.
  • Actualizar dependencias e imágenes base.

Aplicaciones

  • Variables de entorno para configuración.
  • Secretos fuera del repositorio.
  • Logs sin contraseñas ni tokens.
  • Separar ambiente de desarrollo y producción.

Podman y Kubernetes YAML

Podman permite ejecutar definiciones Kubernetes localmente con podman kube play. Esto lo convierte en una herramienta útil para enseñar la transición entre contenedores locales y orquestación.

# Generar YAML desde un pod local
podman generate kube mi-pod > pod.yaml

# Ejecutar YAML estilo Kubernetes
podman kube play pod.yaml

Ejemplo práctico: API mínima

import express from "express";

const app = express();
app.use(express.json());

const items = [{ id: 1, name: "Item 1" }];

app.get("/health", (_, res) => res.json({ status: "ok" }));
app.get("/items", (_, res) => res.json(items));
app.post("/items", (req, res) => {
  const item = { id: items.length + 1, name: req.body.name };
  items.push(item);
  res.status(201).json(item);
});

app.listen(3000, () => console.log("API running on 3000"));

Errores comunes

  • Usar localhost para conectar la API con la base de datos dentro de Compose.
  • Olvidar exponer o mapear el puerto correcto.
  • Guardar datos importantes dentro del contenedor y no en volúmenes.
  • Subir contraseñas reales al repositorio.
  • Construir imágenes enormes por no usar .dockerignore.
  • No revisar logs antes de decir “Docker no funciona”.

Cierre

Docker

Excelente punto de entrada, ecosistema maduro y flujo de trabajo ampliamente adoptado.

Podman

Alternativa moderna, daemonless, orientada a rootless y con buen puente hacia Kubernetes.

Curso

Usar Docker para la base práctica y Podman para discutir seguridad, compatibilidad y operación profesional.
